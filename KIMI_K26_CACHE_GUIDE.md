## 1. 現行価格 (2026-04 時点)

公式: <https://platform.kimi.ai/docs/pricing/chat-k26>

| 項目 | 単価 (per 1M tokens) | 備考 |
| --- | --- | --- |
| Input (Cache Hit) | **$0.16** | キャッシュに既に乗っている prefix 部分 |
| Input (Cache Miss) | **$0.95** | 新規入力、または期限切れの再送 |
| Output | **$4.00** | 通常の生成出力。HIT/MISS と無関係 |
| Context Window | 262,144 tokens | 約 200,000 単語相当 |

**HIT と MISS の差: 約 6 倍** ($0.95 / $0.16 ≒ 5.94)

長い context (リポジトリ全体読み込み等) を扱うほど、HIT/MISS の積算差が効きます。出力 ($4.00) は両方共通なので、節約余地は専ら入力側にあります。

## 2. Moonshot の prompt cache の仕組み

### 2.1 何をキャッシュするか

Moonshot (および DeepSeek / Anthropic) の prompt cache は **「リクエストの先頭から何バイトが過去のリクエストと一致しているか」** を自動判定し、一致した prefix 部分をキャッシュ参照価格 ($0.16/M) で計算する方式。クライアント側で「ここまでキャッシュして」と指示する必要はなく、サーバーが裏で完全自動でやります。

キャッシュキーは概念的に:

```
cache_key = hash(model_id + canonical_serialize(messages_array) + その他パラメータ)
```

### 2.2 prefix 一致とは

会話を 1 ターン進めるとは:

```
turn 1 messages: [sys, u1, a1]
turn 2 messages: [sys, u1, a1, u2, a2]    ← turn 1 の messages がそのまま prefix として残る
turn 3 messages: [sys, u1, a1, u2, a2, u3, a3]
```

ターン 2 の入力を Moonshot に送ると、サーバーは「先頭の `[sys, u1, a1]` 部分は turn 1 のリクエストと完全一致している」と判定し、その部分は HIT 扱い。残りの `[u2]` だけが MISS。

ターン 3 では `[sys, u1, a1, u2, a2]` までが HIT、`[u3]` だけが MISS。

このため、**過去ターンの messages を一切書き換えずに継ぎ足していく限り、入力コストは累積的に最適化** されます。

### 2.3 キャッシュ TTL

Moonshot 公式は具体時間を明示していませんが、業界相場は **15 分〜1 時間** で個々の cache entry が expire します (Anthropic は明示的に 5 分 / 1 時間オプションあり)。

つまり、

- 朝に調査開始 → 昼休憩 1 時間 → 午後再開: **キャッシュ失効、再 MISS** の可能性大
- 短時間に集中して進めた方が安い

## 3. shim 経由はキャッシュに悪影響を与えないことの証明

### 3.1 心配される点

「moonshot-shim が messages を書き換えるなら、書き換え後のハッシュが Moonshot 側のキャッシュキーになる。書き換え方が一定でないと、毎回 MISS になるのでは?」

### 3.2 結論: 影響なし

shim の介入は次の 1 点だけ。

```
assistant && tool_calls.length > 0 && reasoning_content が空
  → reasoning_content = " " を注入
```

この処理は以下の性質を持ちます。

1. **冪等**: 同じ入力に何度適用しても同じ出力。
2. **決定的**: ランダム要素なし。
3. **対象限定**: 介入条件を満たさないメッセージは絶対に触らない。
4. **既存値尊重**: 既に正しい `reasoning_content` がある場合は触らない。

### 3.3 prefix が壊れない実例

```
turn 1 入力 (Cursor が組む):
  [sys, u1, a1(tools, rc=undefined)]
turn 1 入力 (shim 通過後 → Moonshot が見る):
  [sys, u1, a1(tools, rc=" ")]              ← 1 箇所注入

turn 2 入力 (Cursor が組む):
  [sys, u1, a1(tools, rc=undefined), tool1, u2]
turn 2 入力 (shim 通過後):
  [sys, u1, a1(tools, rc=" "), tool1, u2]   ← turn 1 と同じ場所に同じ値を注入
```

ターン 1 で Moonshot が見た `[sys, u1, a1(tools, rc=" "), ...]` の prefix は、ターン 2 でも完全一致。Moonshot 側でちゃんと HIT 判定されます。

### 3.4 やってはいけない shim 改造

placeholder を毎回違う値にすると prefix が壊れて全 MISS になります。

```javascript
msg.reasoning_content = `cache-buster-${Date.now()}`;  // 絶対 NG
msg.reasoning_content = Math.random().toString();       // 絶対 NG
msg.reasoning_content = `turn-${msgIndex}`;             // 一見 OK だが NG
                                                        // (同じ位置のメッセージでも
                                                        //  別会話と被らないとは限らない)
```

現在の `server.js` は固定値 `" "` (半角スペース 1 文字) なので問題ありません。改造の際は固定値を維持してください。

## 5. Cursor 上で具体的にやるべきこと

### 5.1 重い context は最初に一度だけロードする

Agent モードを開いた最初の質問で、その後の質問の前提となる context を全部読ませる。

```
このリポジトリ全体を理解してください。
特に modules/, extensions-builtin/, scripts/ の構造を把握し、
主要なエントリポイントと依存関係を要約してください。
```

- 1 回目: 全 MISS で $0.10 前後
- 以降の細かい質問: ほぼ全 HIT で $0.02/ターン前後

「最初に重く、後は軽く」の構造を意図的に作るのが鍵。

### 5.2 同じスレッドで畳み掛ける

関連する質問はすべて同じ Agent スレッドに継ぎ足す。

```
[Today]
  ▼ A1111 リファクタ調査                ← このスレッドを使い続ける
    ・最初の重いロード
    ・「launch_utils.py の役割は?」      ← HIT で安い
    ・「import_hook.py との関係は?」     ← HIT で安い
    ・「これを修正したい」               ← HIT で安い
```

`+ New Agent` を押した瞬間に過去のキャッシュは無効化扱い。

### 5.3 論理的に区切れたら要約して新スレッドへ

タスクが切り替わるタイミングで、前スレッドに要約させる。

```
ここまでの調査結果と未解決事項を、新しいスレッドに引き継げる形で
箇条書きで要約してください。
```

その箇条書き (例: 5,000 tokens) だけを新スレッドの先頭に貼る。

- 何もしないで新スレッド: 初回 MISS が 100,000 tokens
- 要約引き継ぎ: 初回 MISS が 5,000 tokens
- **20 倍の差**

### 5.4 集中して短時間に進める

キャッシュ TTL は 15 分〜1 時間と推測されます。

- ✅ 良い: 「19:00〜21:00 に集中して 30 ターン進める」
- ❌ 悪い: 「19:00 に 5 ターン → 23:00 に 5 ターン」(後半は全 MISS の可能性)

長期間放置するときは、再開時の「最初の質問が重い」ことを覚悟する。

### 5.5 1 スレッド = 1 モデルで固定

途中でモデルセレクタから別モデル (gpt-5 など) に切り替えると、そのスレッドの過去ターンは別モデル分のキャッシュとは別物になります。

- ✅ 「`kimi-k2.6` で開始したスレッドは、最後まで `kimi-k2.6`」
- ❌ 「途中で gpt-5 に切り替えて比較してみる」 → 切り替え後は全 MISS

## 6. やってはいけないこと

### 6.1 一問一答で `+ New Agent` を連打

```
質問1 → New Agent     ← 100k MISS = $0.095
質問2 → New Agent     ← 102k MISS = $0.097
質問3 → New Agent     ← 104k MISS = $0.099
...
```

10 質問で **$1 越え**。同じスレッドで継続なら $0.27 で済むところを 4 倍払うことになります。

### 6.2 無関係なタスクを同じスレッドに混ぜる

```
[A1111 調査スレッド]
  ・A1111 のリファクタ質問 x5
  ・突然「ComfyUI のノード作成方法は?」  ← context が膨張
  ・また A1111 の質問
  ・「Stable Diffusion の理論を説明して」 ← さらに膨張
```

context が増え続けて 262k 上限に近付くと、Cursor が **古い履歴を勝手に切り詰めて要約** することがあります。これにより prefix が変化してキャッシュが大規模に破壊されます。

論理的に区切れたら別スレッドへ。

### 6.3 Cursor 本体のアップデート直後に大規模質問

Cursor のシステムプロンプトはバージョンアップで変わります。システムプロンプトが変わると prefix の最先頭が変わるため、**過去の全キャッシュが無効化**。

Cursor 更新直後の重い質問は避ける、もしくは 1〜2 日待って前のシステムプロンプトを使い続けるユーザーが多い状態で安定させてから取り掛かる。

### 6.4 messages を書き換える別 proxy を挟む

LiteLLM 等の汎用プロキシを shim と直列に挟む場合、そちらが messages を再シリアライズしたり、キーの順序を変えたりすると、ハッシュが変わってキャッシュ MISS。

shim 単独で使うのが安全。

### 6.5 shim の placeholder を時刻依存値にする

§3.4 参照。固定値 `" "` 厳守。

## 7. 見落としやすい落とし穴

### 7.1 Cursor の自動 context (`@codebase`)

Cursor が裏でやっている embedding 検索は Cursor 側のローカル処理で、Moonshot に到達するのは「検索結果として選ばれたファイル断片」のみ。これも messages 配列に積まれます。

- 毎ターン同じファイルが選ばれる質問パターン → prefix 一致で HIT
- 毎ターン違うファイルが選ばれる質問パターン → prefix が変わって MISS 増

つまり「同じ題材を掘り下げる」会話の方が、自動 context 選択も安定して HIT しやすい。

### 7.2 `@file` で明示的にファイルを添付

`@file:foo.py` のようにユーザーが明示的にファイルを context に入れた場合、それは messages の中に固定で入ります。同じファイルを次のターンでも `@file` し続ければ prefix 一致で HIT。

ターンによってファイル添付を入れたり外したりすると、prefix が変動して MISS。

### 7.3 system prompt の中に時刻が入っているケース

Cursor のシステムプロンプトに「現在時刻: 2026-04-25 12:54」のような時刻情報が含まれていると、ターンごとに先頭が変化して **全ての prefix が無効化**。

これは Cursor 側の実装次第で、ユーザー側で制御不可。Cursor のバージョンによっては時刻を含めない設計のものもあります。

### 7.4 stream=true / stream=false の混在

`stream` パラメータがリクエスト本体に含まれる場合、その値の違いでキャッシュキーが変わる可能性があります。Cursor は基本的に `stream=true` で固定なので、通常意識不要。

### 7.5 tools 定義の差し替え

tools 配列 (model に提供する関数定義のリスト) が messages とは別に送られますが、これも prefix の一部として cache key 計算に入ります。Cursor が会話途中で tools 定義を変えると (拡張機能の有効化等)、キャッシュ MISS。

### 7.6 同時並行スレッド

同じ context を複数スレッドで同時並行で走らせても、キャッシュは共有されます (Moonshot 側で同 API キーの全リクエストが同じキャッシュ空間)。ただし、初回の 1 つは MISS で、2 つ目以降が HIT になるタイミング保証はありません (キャッシュ書き込みに時差あり)。

## 8. キャッシュが効くか効かないかの判定フロー

```
[新しいターンのリクエストを送ろうとしている]
        │
        ▼
  前のターンの messages 配列の先頭部分が、
  今回のリクエストの先頭部分と完全一致しているか?
        │
   ┌────┴────┐
   │ YES     │ NO
   ▼         ▼
  HIT        MISS
   │         │
   ▼         ▼
 $0.16/M    $0.95/M
```

「完全一致」の判定はバイトレベル。1 文字でも違えば MISS。

### 完全一致を保つために必要なこと

| チェック項目 | OK | NG |
| --- | --- | --- |
| 同じスレッド (Cursor の Agent スレッド) | ✅ | `+ New Agent` で別スレッド |
| 同じモデル | ✅ | 途中切替 |
| 同じシステムプロンプト | ✅ | Cursor アップデート直後 |
| 同じ tools 定義 | ✅ | 拡張機能の有効化中 |
| 同じ shim | ✅ | shim を別実装に置換 |
| placeholder 固定値 | ✅ | 時刻 / ランダム化 |
| 履歴の途中改変なし | ✅ | Cursor が context を圧縮 |
| TTL 内 (1 時間以内に再送) | ✅ | 数時間以上の放置 |

## 9. 実例ワークフロー

### 9.1 大規模リポジトリの調査タスク (3 時間集中作業)

```
[19:00] スレッド「A1111 調査」を新規作成
        ↓
[19:00] 「リポジトリ全体を理解して、modules/ の構造を要約してください」
        → 100k MISS = $0.10
        ↓
[19:05〜21:30] 同じスレッドで 25 質問を継ぎ足し
        → 各 ~$0.025、計 $0.625
        ↓
[21:30] 「ここまでの調査結果と次タスクの引き継ぎを箇条書きで」
        → ~$0.025
        ↓
[21:30] スレッド終了

合計: 約 $0.75 (大規模調査としては破格)
```

これを「1 質問ごとに新スレッド」でやると約 $3 〜 $4。

### 9.2 翌日に続きをやる場合

```
[翌日 10:00] 前日のスレッドを再開して質問
        → キャッシュ TTL 切れている可能性大
        → 100k 再送 = MISS = $0.10 が再度発生
```

これを避けるなら:

```
[翌日 10:00] 新スレッドを作成
        → 前日の最後に作った要約 (5k tokens) を貼る
        → 「これを前提に、次の質問: ...」
        → 5k MISS = $0.005 + 質問 = ほぼ無料に近い
```

### 9.3 短い単発質問 (5 ターン未満)

```
質問が 1〜3 個で終わる予定
        → キャッシュ最適化を考えなくて良い
        → 普通にやる
        → どっちみち $0.05〜$0.15 程度
```

過剰最適化は不要。**長く重いタスクの時だけ意識すれば十分**。

## 10. 一行まとめ

> **「重い context は最初に一度ロード」 → 「同じスレッドで畳み掛ける」 → 「区切りで要約して新スレッドへ」**

この 3 動作をルーティンにするだけで、典型的に **3〜5 倍の節約**。

shim は完全に裏方で、上の動作とは独立に勝手に動いており、ご意識いただくのは Cursor 上の「スレッドの使い方」だけで十分でございます。

## 付録: 価格改定があった場合の試算式

公式が価格を改定した場合、本ガイドの試算は以下の式で更新できます。

### ケース A (同スレッド継続, N ターン)

```
total_A = (P_miss × C0) + Σ_{t=2}^{N} (P_hit × cumul_{t-1} + P_miss × Δ)

P_miss   : MISS 単価 ($/token)
P_hit    : HIT 単価 ($/token)
C0       : 初回 context size (tokens)
Δ        : 1 ターンごとの新規追加 tokens
cumul_t  : ターン t 終了時点の累計入力 = C0 + (t-1)×Δ
N        : 総ターン数
```

### ケース B (毎回新スレッド, N ターン)

```
total_B = Σ_{t=1}^{N} P_miss × cumul_t
```

### 差額

```
saving_ratio = total_B / total_A
            ≒ P_miss / P_hit (N が大きく Δ << C0 のとき)
```

つまり HIT/MISS の単価比 (現在 ≒ 6 倍) が、長期スレッドにおける節約上限。
